# 運営確認事項（お客様・ビジネス向け）

Hyperpool を公開・運用する前に、**お客様（運営・プロダクト・法務）** が判断・合意すべき項目だけをまとめたものです。

- 技術用語・コードの話は省いています
- 各項目は **質問 → いまの提案 → 決定欄** の形式
- 実装・設定の詳細は開発側の [運営確認事項.md](./運営確認事項.md) を参照

> プロダクト概要: [product-overview.md](../product-overview.md)

---

## 使い方

1. 打ち合わせでセクションごとに読み上げ、**決定欄**に記入
2. 「いまの提案」と異なる場合は、**いつまでに・誰が**決めるかもメモ
3. 決定内容は社内 Wiki / 契約書 / ユーザー向け説明に反映

---

## よくある質問（運営からの確認）

### Q1. 報酬分配は 67:33 ですか？

**結論: はい。運営 33% / ユーザー還元 67% です。**

| 段階 | 内容 | 割合 |
|------|------|------|
| ① Project X で手数料を collect | USDC + HYPE | 100% |
| ② 運営へ送金 | USDC・HYPE それぞれ | **33%** |
| ③ ユーザー還元（Cashdrop） | USDC のみ、毎朝分配 | **67%** |
| ④ HYPE の残り | Vault 内に残留（全員の評価額に反映） | **67%** |

**補足:**
- 画面の APR 目安: 参考 APY 75% × 67% ≒ **50.25%**
- 紹介報酬（15% / +10%）は **67% プールの中**での再配分（運営 33% とは別）

---

### Q2. レンジは上 +10% 固定、下は選べる？

**結論: 全員 +10% / −30% 固定です。ユーザーによる選択はできません。**

| 項目 | 設定 |
|------|------|
| 上限（上） | **+10%** 固定 |
| 下限（下） | **−30%** 固定 |
| ユーザー選択 | **不可**（全員同じレンジ） |

---

### Q3. リバランスは上に抜けたら？下に抜けたら？

**上でも下でも同じ処理です。** 現在の市場価格を中心に +10% / −30% で組み直します（約 6 時間ごと）。

| 状況 | レンジ外のあいだ | リバランス時 |
|------|------------------|--------------|
| 上に抜けた | 手数料ほぼゼロ・資産は HYPE 寄り | 現在価格中心で +10 / −30 に再設定 |
| 下に抜けた | 手数料ほぼゼロ・資産は USDC 寄り | 現在価格中心で +10 / −30 に再設定 |

## 1. 収益の分け方（最重要）

| # | 確認したいこと | いまの提案 | 決定欄 |
|---|----------------|------------|--------|
| 1-1 | Project X の取引手数料を **運営 33% / ユーザー 67%** で分ける方針でよいか？ | 67% は毎朝 USDC でユーザーに還元（Cashdrop） | |
| 1-2 | Project X では手数料が **WHYPE/USDC 両方**で発生する点を理解できるか？ Hyperpool は **harvest 時に USDC へ統一**してから 67% を Cashdrop する | スワップ slippage 50bps（owner 変更可） | |
| 1-3 | **運営の受取ウォレット**（手数料 33% の送金先）はどのアドレス／どの組織が管理するか？ | デプロイ時は開発側ウォレット | |
| 1-4 | 画面に出す **APR・収益の見せ方**（「参考 APY × 約 67%」など）はユーザーに誤解を与えないか？ | 保証 APR ではなく参考値と明記 | |
| 1-5 | **ブリッジ（他チェーンからの入金）**に Hyperpool 独自の手数料を載せない（0%）方針でよいか？ | 現状 0% | |
| 1-6 | ブリッジ時の **スリッページ許容**の初期値 0.5% でよいか？ | ユーザーが設定で変更可能 | |

**注意:** 1-1 の比率を変える場合は、**公開前に**開発側と相談が必要です（公開後は変更が難しい設計です）。

## 2. 毎朝の USDC 還元（Cashdrop）

| # | 確認したいこと | いまの提案 | 決定欄 |
|---|----------------|------------|--------|
| 2-1 | 手数料の集計（harvest）を **毎朝 JST 7:00** に実行する運用でよいか？ | 日本時間の朝 | |
| 2-2 | ユーザーに **claim 操作をさせず自動送金**する方針でよいか？ | 毎朝 JST 7:00 に対象ウォレットへUSDC送金 | |
| 2-3 | ユーザー側の請求漏れリスクをなくす方針でよいか？ | Claimボタンなし・期限なし・運営処理で自動送金 | |
| 2-4 | 送金失敗時の確認・再実行手順を運営側で持つか？ | 配分IDで二重送金を防止しつつ、失敗Txは運営が確認 | |
| 2-5 | Cashdrop を **毎日自動**で回す方針でよいか？ | 自動 cron 前提 | |

**ユーザー向け説明の例:** 「毎朝 7 時頃に前日分の手数料を集計し、対象ウォレットへUSDCを自動送金します。ユーザー側で請求ボタンを押す必要はありません。」

## 9. 数字の早見表（合意のたたき台）

お客様が「変更したい」と言える主な数字です。**公開後に変えにくいもの**には ⚠️ を付けています。

| 項目 | いまの提案 | 変更のしやすさ |
|------|------------|----------------|
| 運営 vs ユーザー（USDC 手数料） | **33% / 67%** | ⚠️ 公開前のみ相談 |
| HYPE 手数料のユーザー分 | harvest 時 USDC 換金 → 67% Cashdrop | スワップ slippage に注意 |
| 紹介者報酬 | 15%（67% プール内） | ⚠️ 公開前のみ相談 |
| 被紹介者ブースト | +10%（67% プール内） | ⚠️ 公開前のみ相談 |
| LP 上限 | +10% 固定（全員） | 変更可能（要開発） |
| LP 下限 | −30% 固定（全員・選択不可） | 変更可能（要開発） |
| 対象プール | WHYPE/USDC 0.05% | 公開後も変更可能（要開発作業） |
| Cashdrop 時刻 | harvest / 自動送金 7:00 | 運用・UI で調整可 |
| ブリッジ手数料（Hyperpool） | 0% | 開発作業で変更可 |
| デモ表示 | Testnet のみ / Mainnet 前に削除 | 必須判断 |
