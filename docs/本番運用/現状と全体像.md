# 現状と全体像

運営が環境を立ち上げるとき、**何がどこで動くか**を最初に押さえてください。

---

## 2 層構造

Hyperpool の「稼働」は **フロント** と **オンチェーン** の両方が揃って初めてユーザー体験が完成します。

```
┌──────────────────────────────────────────────────────────────┐
│ レイヤー 1: フロントエンド（Vercel）                          │
│ ・ブラウザ + MetaMask から操作する GUI                        │
│ ・コントラクトはここではデプロイしない                        │
│ ・設定: NEXT_PUBLIC_DEFAULT_CHAIN_ID, RPC, WalletConnect 等   │
└────────────────────────────┬─────────────────────────────────┘
                             │ JSON-RPC（読取・TX 送信）
┌────────────────────────────▼─────────────────────────────────┐
│ レイヤー 2: HyperEVM スマートコントラクト                     │
│ ・ Chain 998 = Testnet / Chain 999 = Mainnet                  │
│ ・ Factory, Router, Pair, Vault, Points, Airdrop 等           │
│ ・ Swap には Pair プールへの流動性（kHYPE + USDC）が必須       │
└──────────────────────────────────────────────────────────────┘
```

**Vercel だけデプロイしても DEX は動きません。** オンチェーンの初期化（特に **流動性**）が別途必要です。

---

## Swap を動かすために「何をどこに入れるか」

よくある誤解: 「Router コントラクトに USDC を送れば Swap できる」

**正しくは:** Uniswap V2 型 AMM の **Pair プール** に、両方のトークンを **`addLiquidity`** で入れます。

| トークン | 役割 | 誰が用意するか |
|---------|------|----------------|
| **HYPE** | ガス代（全 TX） | 運営ウォレット・ユーザー各自 |
| **kHYPE**（WHYPE `0x5555…`) | プール片側 | 運営が wrap、またはユーザー保有 |
| **USDC** | プール片側 | Hyperliquid から EVM ブリッジ、またはユーザー保有 |

流動性が入ると:

- Swap タブでレート表示・スワップ TX が可能
- スワップ手数料が発生 → Points が記録される（プールが authorize 済みであること）

流動性が **0** のとき:

- UI は開けるがレートは `—`
- スワップは失敗または revert

## 機能ごとに運営がやること（一覧）

| 機能 | コントラクトに資金を「入れる」？ | 運営の作業 |
|------|----------------------------------|------------|
| **画面表示** | 不要 | Vercel デプロイ・env 設定 |
| **同一チェーン Swap** | **Pair に kHYPE + USDC（LP）** | 流動性シード or ユーザー LP 待ち |
| **LP / Vault** | ユーザー or 運営が預ける | Vault はデプロイ済み。必要なら Admin で pause 等 |
| **Points** | 預け入れではない（手数料連動） | デプロイ時 `authorizePool` 済み。スワップが発生すること |
| **Cashdrop** | **MerkleAirdrop に USDC** を `fund` | `setMerkleRoot` + `fund` + `airdropEntries` を JSON に同期 |
| **Affiliate** | 不要 | ユーザーがコード登録 |
| **Li.FI クロスチェーン** | 別経路 | **Testnet 998 は非対応**。本番 999 向け |
| **Admin** | 不要 | Preview/ローカルで有効化、owner ウォレット |

## Testnet の現状（2026-06 時点・オンチェーン確認済み）

`contracts/deployments/998.json` と RPC 読取の結果:

| 項目 | 状態 |
|------|------|
| コントラクトデプロイ | ✅ **済み**（全アドレスにコードあり） |
| `deployed: true` | ✅ |
| Liquidity Vault (Phase 3) | ✅ デプロイ済み |
| プール Points authorize | ✅ 済み |
| **プール残高（kHYPE / USDC）** | ❌ **0 / 0** |
| Merkle Cashdrop | ❌ ルート未設定 |
| `airdropEntries`（フロント用） | ❌ JSON に未記載 |

→ **「コントラクトはデプロイ済み」という理解は正しい。**  
→ **「ユーザーが Swap できる状態」にはまだなっていない**（流動性シードが未実施）。

## 運営ウォレットに常に必要なもの

| 用途 | トークン |
|------|---------|
| デプロイ・Admin TX | HYPE（ガス） |
| 流動性シード | HYPE + kHYPE + USDC |
| Cashdrop fund | USDC（Airdrop コントラクトへ） |

Testnet では:

- ガス: [Hyperliquid Testnet Drip](https://app.hyperliquid-testnet.xyz/drip)
- USDC / wrap: `scripts/testnet-post-deploy.mjs` が手順を自動化
