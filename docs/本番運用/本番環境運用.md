## 本番公開の流れ（概要）

### 1. コントラクトデプロイ

```bash
source scripts/testnet-env.sh   # または .env.mainnet
./scripts/deploy-mainnet.sh
node scripts/sync-abi.mjs
```

- big blocks 有効化、十分な HYPE ガス
- `contracts/deployments/999.json` をコミット
- HyperEVM RPC が receipt 待機で `invalid block height` / `invalid block range` を返すことがある。その場合はオンチェーン code を確認し、必要なら手動で以下を設定:

```bash
cast send <adapter> 'setVault(address)' <vault> ...
cast send <adapter> 'setPool(address)' 0x6c9A33E3b592C0d65B3Ba59355d5Be0d38259285 ...
cast send <vault> 'setSwapRouter(address)' 0x1EbDFC75FfE3ba3de61E7138a3E8706aC841Af9B ...
cast send <airdrop> 'setVaultShareToken(address)' <vault> ...
```

設定確認:

```bash
cast call <adapter> 'vault()(address)' --rpc-url https://rpc.hyperliquid.xyz/evm
cast call <adapter> 'pool()(address)' --rpc-url https://rpc.hyperliquid.xyz/evm
cast call <vault> 'swapRouter()(address)' --rpc-url https://rpc.hyperliquid.xyz/evm
cast call <airdrop> 'vaultShareToken()(address)' --rpc-url https://rpc.hyperliquid.xyz/evm
```

### 2. Vault deposit + Project X 代理 LP

Mainnet では運営が LP 原資を用意して自前プールを作るのではありません。ユーザーが **Position** タブから Vault に USDC / HYPE を預けると、Vault が `ProjectXAdapter` 経由で既存の Project X WHYPE/USDC NPM ポジションに代理投入します。

- 初回 depositor が Vault shares を受け取り、1000 shares は inflation 対策として `0xdEaD` にロック
- 運営の必須資金はデプロイ・keeper・daily-rewards 用の **HYPE ガス**
- 事前 smoke のために運営ウォレットで少額 deposit することは任意。本番LP原資としては扱わない
- `testnet-post-deploy.mjs` は 998 専用。Mainnet では通常実行しない
- 片側入金（USDCのみ / WHYPEのみ）は Vault が一部を Project X router で反対側へswapし、両側流動性として NPM mint
- 初回レンジは Adapter が Project X pool `slot0` の現在価格を読み、+10% / -30% の範囲で作成

### 2.1 Mainnet smoke 手順

1. 少額 USDC（例: `0.07 USDC`）を deployer / 検証ウォレットに用意
2. `depositUSDC(amount, receiver)` を実行
3. `adapter.positionTokenId()` が 0 以外になることを確認
4. `adapter.tickLower() < Project X currentTick < adapter.tickUpper()` を確認
5. `harvestFees()` を実行し、tx が成功することを確認
6. Project X pool / oracle価格の5%以内で `rebalance(refPrice)` を実行
7. `positionTokenId` が更新され、rebalance後もレンジ内であることを確認

2026-06-25 最終 smoke では `0.07 USDC` deposit、harvest、+0.5% 価格での rebalance が成功。少額かつ直後 harvest のため `pendingUserRewards` 増分は `0`。

### 3. Keeper / 日次 cron

日次 Cashdrop はユーザー claim ではなく、`daily-rewards.mjs` が毎日1回起動して対象者へUSDCを自動送金します。

```bash
DEPLOYMENT_CHAIN=999 node scripts/keeper-rebalance.mjs
DEPLOYMENT_CHAIN=999 node scripts/daily-rewards.mjs
```

正式公開前は Mac cron ではなく VPS / GitHub Actions へ移行し、失敗通知（Slack / メール等）を付けることを推奨します。最低限、以下を日次で確認します。

- `keeper-rebalance.mjs` が成功し、価格乖離や Project X slippage で止まっていない
- `daily-rewards.mjs` が `harvestFees`、`pullPendingRewards`、`distributeRewards` を完了
- 非ゼロ手数料が発生した日には operator 33% / user 67% の配分が想定通り

## 本番で「コントラクトに入れる」もの

| 目的 | 何をどこに |
|------|-----------|
| Vault LP | ユーザーが HyperpoolVault へ USDC / HYPE deposit |
| Cashdrop | harvest 後 `pullPendingRewards` → MerkleAirdrop → `distributeRewards` でユーザーへ自動送金 |
| ガス | deployer / owner / keeper ウォレットに HYPE |

運営が本番LP原資を常時用意する必要はありません。Cashdrop も Project X LP 手数料 harvest から生まれる 67% ユーザー分を自動送金します。

## リリース判定

| 区分 | 判定 | 理由 |
|------|------|------|
| 内部ベータ / 限定公開 | GO | Mainnet deposit、Project X LP mint、harvest tx、rebalance が成功 |
| 一般向け正式公開 | 条件付きGO | 非ゼロ手数料の Mainnet harvest と cron 監視の確認後が望ましい |

残リスク:

- 極小LPでは即時 harvest が `0` になりやすい。十分なLP規模または時間経過後に非ゼロfeeを確認する
- owner / keeper / operator のホットウォレット運用は最小期間にし、マルチシグ化を検討
- 旧初回 smoke の約 `1 USDC` は旧Vault withdrawバグにより回収不能。最終版では修正済み
