# 運営確認事項（開発・DevOps 向け）

Hyperpool を Testnet / Mainnet で運用する前に、**開発・DevOps** が確認・実装すべき項目をまとめたものです。

- **コード上の現行デフォルト**を記載しています（2026-06 時点のリポジトリ）
- 各項目は **確認質問 → 現状 → 決定欄** の形式
- **お客様・ビジネス判断**だけに絞った版 → [運営確認事項_お客様向け.md](./運営確認事項_お客様向け.md)
- 決定後は [チェックリスト.md](./チェックリスト.md) と併用してください

> 開発詳細: [architecture.md](../architecture.md) · [product-overview.md](../product-overview.md) · [admin-guide.md](../admin-guide.md)

---

## 使い方

1. 運営ミーティングでセクションごとに読み上げ、**決定欄**に回答を記入
2. 変更が必要な数値は **再デプロイ要否** を確認（多くの BPS は immutable）
3. 決定内容を Notion / Issue / 社内 Wiki に転記し、本ファイルの決定欄も更新

## A. 収益分配・手数料（最重要）

| # | 確認質問 | コード上の現行値 | 再デプロイ | 決定欄 |
|---|----------|------------------|------------|--------|
| A-1 | **Project X 手数料の運営 vs ユーザー分配比率**は 33% / 67% で確定か？ | USDC: 33% → `operatorWallet`、67% → `pendingUserRewards` → Cashdrop 自動送金 | **immutable**（`operatorFeeBps` に setter なし） | |
| A-2 | **HYPE 手数料**の扱いは理解・合意済みか？ | harvest 時に Swap Router で USDC 換金 → **33/67 とも USDC**（`convertHypeFeesToUsdc=true`） | owner 可変 | |
| A-3 | **運営ウォレット（operatorWallet）**はどのアドレスか？ | デプロイ時 = deployer（constructor で 0 なら owner） | `setOperatorWallet()` で変更可 | |
| A-4 | **APR 表示**の説明は正確か？ | 参考 APY × 67% を脚注表示（`constants.ts` `PROJECT_X_POOL`） | UI のみ | |
| A-5 | **Li.FI ブリッジ手数料**は 0% でよいか？ | API `fee=0` 固定（`lifi/quote/route.ts`） | env / コード変更 | |
| A-6 | **ブリッジスリッページ**のデフォルトは 0.5% でよいか？ | 50 BPS（`SettingsModal.tsx`、localStorage で変更可） | UI のみ | |

**補足:** `OPERATOR_FEE_BPS` 環境変数は `daily-rewards.mjs` で **ログのみ**。オンチェーン比率は変わりません。

## B. Cashdrop（日次 USDC 還元）

| # | 確認質問 | コード上の現行値 | 再デプロイ | 決定欄 |
|---|----------|------------------|------------|--------|
| B-1 | **harvest 実行時刻**は JST 7:00 でよいか？ | cron / GitHub Actions `0 22 * * *` UTC = JST 7:00 | cron 設定のみ | |
| B-2 | **UI 請求ウィンドウ**を廃止するか？ | Claimボタンなし。ユーザー操作不要 | UI のみ | |
| B-3 | **自動送金Tx**は JST 7:00 に実行するか？ | `daily-rewards.mjs` が `distributeRewards` を実行 | cron ロジック | |
| B-4 | **配分IDによる二重送金防止**は必須か？ | `distributionExecuted[distributionId]` で同一配分の再実行を拒否 | コントラクト | |
| B-5 | **送金失敗時の運用**は？ | Tx失敗ならAirdrop残高・対象者リスト・distributionIdを確認して再実行 | 運用判断 | |
| B-6 | **自動送金フロー**は cron 一本化か？ | `daily-rewards.mjs` → git push → Vercel redeploy | 運用判断 | |
| B-7 | **対象者スナップショット**は意図どおりか？ | `vaultShareHolders` と紹介情報からJST 7:00時点の配分を構築 | スクリプト | |
| B-8 | **Admin 手動 Merkle** は廃止済みでよいか？ | Admin は pause / auto payout 状態確認のみ | UI のみ | |

**JST 7:00 の理由:** harvest を朝 7 時に実行し、その時点の Vault シェアと紹介情報をもとに対象ユーザーへUSDCを自動送金します。ユーザー側の claim 操作・請求漏れ・期限切れはありません。

**Testnet 注意:** 998 では本物 USDC 手数料 harvest が不完全な場合あり → `testnet-daily-rewards-smoke.mjs` で代替検証。Mainnet では `daily-rewards.mjs` のみ使用。

## I-補足. Mainnet デプロイ / smoke メモ（2026-06-25）

| 項目 | 結果 |
|------|------|
| Deployer | `0x0196f2949FbcE973d54d2047E3B8bfAde06e8ceC` |
| デプロイ実ガス | `0.00089856648 HYPE`（8 tx receipts 合計） |
| デプロイ後 Vault | `0xdec7711864c64d6d98ab61e5e28d1a2fd76f9c87` |
| デプロイ後 Adapter | `0x127203e9b55620f9ca6ba5056e34cbbd0c9ddf92` |
| 本番 Vercel | `https://hyper-evm-ten.vercel.app`（`NEXT_PUBLIC_DEFAULT_CHAIN_ID=999`） |
| HYPE smoke | `0.01 HYPE` wrap + approve までは成功、`depositHYPE` は送信前 simulation で revert |
| USDC smoke | spot `1.0 USDC` を HyperEVM へ bridge し、Vault `depositUSDC(1.0)` 成功 |
| Vault shares | deployer `999000` / totalSupply `1000000`（初回 dead shares `1000`） |
| Project X NFT | `positionTokenId=504840` |
| smoke後 totalAssets | `0.999999 USDC` |
| smoke harvest | `harvestFees` は成功、ただし `pendingUserRewards` delta は `0` |

**重要:** 現デプロイでは初回 LP レンジが参照価格 `42 USDC/HYPE` 近辺で作られた一方、Mainnet 実勢価格は約 `67 USDC/HYPE` 付近だったため、発行された Project X NFT はレンジ外。預入・NFT発行は確認できたが、このままでは手数料獲得 smoke は未達。Mainnet公開前に、初回/追加入金時に現在価格へ合わせて両側流動性を作る処理（片側入金の一部swap、または別のLP組成方式）を修正して再デプロイすること。

| 項目 | 結果 |
|------|------|
| 最終 Vault | `0x2DB5FCfC0c9Eed612A544B99C9097FbBC0Cf502d` |
| 最終 Adapter | `0x3193AfC8FBEEDB0c2ee5B2F9Ed287579c2aa1796` |

| 本番 Vercel | `https://hyper-evm-ten.vercel.app`（既存 `hyper-evm` project へ prod deploy / alias 済み） |
| 回収 | 旧初回 smoke の約 `1 USDC` LP は旧 `withdrawProRata` バグにより回収不能。以後の修正版 smoke 資金は redeploy 前に withdraw 回収済み。 |
| 修正1 | Adapter が Project X pool `slot0` から現在価格を取得し、初回LPレンジを現在価格基準で作成。 |
| 修正2 | Vault の片側入金は一部を Project X router で反対側へswapし、両側流動性としてNPMへmint。 |

| 最終 USDC smoke | `depositUSDC(0.07)` 成功。shares `69000` / totalSupply `70000`（dead shares `1000`）。 |
| Project X NFT | deposit後 `504910`、rebalance後 `504912`（tokenId 変更確認） |

**残リスク:** 実Mainnetで非ゼロ手数料のCashdrop反映を確認するには、十分なLP規模または時間経過後のProject X取引量が必要。今回の `0.07 USDC` smokeではLPシェアが極小のため、即時harvestで非ゼロfeeは期待できない。
